# Pathfinder — техническая спецификация (v1)

**Дата:** 2026-07-21
**Платформа:** macOS (Apple Silicon + Intel)
**Стек:** SwiftUI + встроенный поисковый движок `fff` через C ABI
**Принцип:** максимальная простота использования. Один экран, минимум настроек на виду, всё сложное — свёрнуто.

---

## 1. Цель

Настольное приложение для macOS, которое ищет по текстовым файлам (исходники, конфиги, данные) и показывает результаты в удобном для чтения виде. Поддерживает поиск, замену с отменой, предпросмотр файла целиком, действия над файлами результата и интеграцию с Finder.

Прообраз — dnGrep / Pathfinder. Данный документ намеренно сокращает функционал прообраза в пользу простоты (см. §9 «Отложено»).

---

## 2. Поисковый движок: `fff`

Ядро поиска не пишем сами — используем открытый движок [`fff`](https://github.com/dmtrKovalenko/fff) (Rust, лицензия MIT).

**Почему `fff`:**
- Резидентный in-memory индекс, повторные запросы < 10 мс.
- Три режима поиска из коробки: plain / regex / fuzzy.
- SIMD-ускоренный матчинг.
- Фоновый файл-watcher — индекс обновляется инкрементально при изменении файлов.
- Осведомлённость о git-статусе (modified / staged / untracked) — используем для будущих фильтров.
- Экспортирует стабильный C ABI (`libfff_c`), что позволяет вызывать его из Swift.

**Что `fff` НЕ покрывает** (и потому вырезано из v1): XPath, byte-sequence, phonetic как отдельный режим. Fuzzy-режим `fff` частично закрывает потребность «нестрогого» поиска.

### 2.1. Способ подключения (решение)

**Выбрано: C ABI (`libfff_c`) + Swift bridging header.**

- `libfff_c` — стабильный C-интерфейс, поставляется проектом `fff`.
- Собираем статическую/динамическую библиотеку, подключаем в Xcode через bridging header.
- Вызовы in-process, без IPC и подпроцессов.

Альтернатива (не выбрана): свой Rust-враппер поверх крейта `fff-search` с собственным C-экспортом под нужды UI. Гибче, но требует Rust-тулчейна в сборке проекта. Оставлено на случай, если стандартного C ABI не хватит.

## 3. Пользовательский интерфейс

Один главный экран. Плотность dnGrep сознательно снижена.

```
┌───────────────────────────────────────────────────────┐
│  [📁 Папка ▾]            Text ○   Regex ○   Fuzzy ○     │
│  [ Найти:    _______________________________ ]   🔍     │
│  [ Заменить: _______________________________ ] [Replace]│
│  ▸ Фильтры:  include *.kt,*.json   exclude build/       │
│  ▸ Контекст: до [2 ▾]   после [2 ▾]                     │
├──────────────────────────┬────────────────────────────┤
│ Результаты               │  Предпросмотр файла          │
│                          │                              │
│ file.kt (3)              │   ...содержимое файла         │
│    40  fun search(q) {   │   с подсветкой всех           │
│    41    val o = def()   │   совпадений                  │
│ ▸  42    engine.run(q,o) │                              │
│    43  }                 │                              │
│ other.json (1)           │                              │
├──────────────────────────┴────────────────────────────┤
│ 198 файлов · 2404 совпадения · 0.3 с                    │
└───────────────────────────────────────────────────────┘
```

### 3.1. Элементы

- **Выбор папки** — кнопка/дропдаун, помнит недавние папки.
- **Режим поиска** — сегментированный переключатель: Text / Regex / Fuzzy.
- **Поле «Найти»** — основной ввод. Дебаунс 200 мс, поиск по мере ввода. Индикатор валидности regex.
- **Поле «Заменить» + кнопка Replace** — опционально; замена применяется к текущим результатам.
- **Фильтры** (свёрнуто по умолчанию): include-glob, exclude-glob, исключить бинарные, лимит размера файла.
- **Контекст** (свёрнуто): строки до/после совпадения, дефолт 2/2, значение 0 = только строка совпадения.
- **Список результатов** — сгруппирован по файлу, у файла счётчик совпадений. Совпадение показывается блоком: контекст приглушён, строка совпадения подсвечена, номера строк слева.
- **Панель предпросмотра** — весь файл с подсветкой всех совпадений; клик по результату скроллит к нужной строке.
- **Статус-бар** — число файлов, число совпадений, время.

### 3.2. Контекстное меню результата (правый клик)

- Показать в Finder (reveal).
- Открыть во внешнем редакторе (Sublime и др.) — на нужной строке, если редактор умеет.
- Копировать путь.
- Move / Copy / Delete файла.

### 3.3. Темы

Светлая/тёмная — через системную тему SwiftUI, без своего кода тем.

## 4. Функционал v1

- Поиск: Text, Regex, Fuzzy.
- Фильтры: include/exclude glob, лимит размера, исключение бинарных.
- Context lines: N строк до/после совпадения (настраивается).
- Результаты сгруппированы по файлу со счётчиком.
- Предпросмотр файла целиком с подсветкой.
- Search-and-replace с undo. Замена только в текстовых файлах (бинарные — read-only).
- Правый клик → показать в Finder.
- Открытие результата во внешнем редакторе.
- Move / Copy / Delete файлов результата.
- Системные темы.

## 5. Архитектура (модули)

Каждый модуль — одна ответственность, тестируется отдельно.

| Модуль | Ответственность | Зависит от |
|---|---|---|
| **FffCore** | Swift-обёртка над C ABI `libfff_c`. Принимает query + опции, отдаёт async-поток результатов. Добирает context lines (±N) из файла. | `libfff_c` |
| **SearchModel** | Состояние поиска (`@Observable`): текущий запрос, режим, фильтры, контекст. Дебаунс ввода, отмена предыдущего поиска. | FffCore |
| **ResultsStore** | Хранит и группирует результаты по файлу, сортировка, текущий выбор. | — |
| **ReplaceEngine** | Применение замены к текстовым файлам, undo-стек (snapshot содержимого до записи). | ResultsStore, FileOps |
| **FileOps** | move / copy / delete / reveal-in-Finder / открыть во внешнем редакторе. Обёртка над `FileManager` и `NSWorkspace`. | — |
| **UI** | SwiftUI-вьюхи: SearchBar, FiltersPanel, ResultsList, PreviewPane, StatusBar. | SearchModel, ResultsStore |

### 5.1. Модель результата

```swift
struct SearchMatch {
    let file: URL
    let lineNumber: Int
    let matchRange: Range<Int>      // диапазон совпадения внутри строки
    let matchLine: String
    let contextBefore: [NumberedLine]
    let contextAfter: [NumberedLine]
}

struct NumberedLine {
    let number: Int
    let text: String
}

struct FileResult {
    let file: URL
    let matches: [SearchMatch]      // matches.count = счётчик у файла
}
```

## 6. Поток данных

**Поиск:**
```
ввод в поле → дебаунс 200мс → SearchModel собирает query+opts
  → FffCore вызывает libfff_c → async-поток совпадений
  → FffCore добирает ±N context lines → ResultsStore группирует по файлу
  → UI рендерит список + статус-бар
```

**Отмена:** новый ввод отменяет предыдущий поиск (Swift `Task` cancellation).

**Замена:**
```
Replace → ReplaceEngine: для каждого текстового файла результата
  снимает snapshot содержимого → пишет замену на диск → пушит в undo-стек
Undo → восстанавливает snapshot(ы) из стека
```

**Действия над файлами:** UI → FileOps → `FileManager` / `NSWorkspace`; после изменения FS — обновление результатов (watcher `fff` подхватит).

## 7. Обработка ошибок

- **Невалидный regex** — не запускать поиск, показать красную подсказку у поля.
- **Нет доступа к папке** (macOS sandbox / права) — запросить доступ через `NSOpenPanel`, security-scoped bookmark; при отказе — сообщение.
- **Файл изменился между поиском и заменой** — сверить snapshot; при расхождении пропустить файл и сообщить.
- **Ошибка записи при замене** — откатить этот файл, продолжить остальные, показать список неуспешных.
- **Бинарный файл в результатах при Replace** — пропустить, пометить как read-only.
- **Крэш/ошибка C-слоя** — оборачивать вызовы `libfff_c`, ошибки транслировать в Swift-`Error`, не ронять UI.

---

## 8. Тестирование

- **FffCore** — фикстуры-файлы, проверка соответствия совпадений и корректности окна context lines для каждого режима.
- **ResultsStore** — группировка/сортировка/выбор на синтетических данных.
- **ReplaceEngine** — замена + undo на временных файлах; сценарий «файл изменился»; сценарий ошибки записи.
- **FileOps** — move/copy/delete во временной директории.
- **SearchModel** — дебаунс и отмена (проверка, что устаревший поиск не мутирует стор).
- **UI** — smoke-тесты вьюх (SwiftUI previews / snapshot по возможности).

## 9. Отложено (не входит в v1)

- XPath-поиск.
- Byte-sequence поиск.
- Phonetic как отдельный режим (частично закрыт fuzzy).
- Bookmarks (сохранённые паттерны).
- Test-window для сложных запросов.
- Save search results.
- Поиск внутри архивов.
- Boolean-операторы, capture-group-поиск и прочие плотные фильтры dnGrep.

Эти пункты можно добавить в следующих версиях; часть (XPath, byte) потребует собственного кода вне `fff`.
